## 1. 단위

- 이름: **크레딧** (Credit)
- 아이콘: 🪙
- 채택 이유:
  - AI 서비스 업계 표준 용어 (OpenAI, Anthropic, Replicate 등) — 사용자 학습 곡선 0
  - 다국어 호환 우수 (credit / クレジット / 积分 / crédito / tín dụng / เครดิต / crédito)
  - 차감 단위·가격 정책을 자유롭게 조정해도 사용자 거부감 적음 (이미 "AI 사용량" 으로 인지됨)
- 기존 보상 단위 **gem (💎)** 과 역할 분리:
  - **gem (💎)**: 적립 — 이미지 발견 보상 등
  - **크레딧 (🪙)**: 결제·소비 — Reading 사진 분석 등

## 2. 충전 옵션

| 결제 금액 | 크레딧 | 평균 페이지 | 1 페이지당 가격 | 할인율 |
|---|---|---|---|---|
| **5,000원** | 100 크레딧 | 50장 | 100원 | — |
| **10,000원** | 300 크레딧 | 150장 | 67원 | **33%↓** |
| **20,000원** | 700 크레딧 | 350장 | 57원 | **43%↓** |

### 업그레이드 인센티브 곡선

- 5천 → 1만: 2배 결제로 3배 페이지 (강한 업그레이드 동기)
- 1만 → 2만: 2배 결제로 약 2.3배 페이지 (추가 할인 인센티브)

## 3. 차감 정책 — 자동·불투명

### 차감 단가 (예시)

| 페이지 텍스트 양 | 차감 크레딧 |
|---|---|
| 짧은 페이지 (~500자 이하) | **1 크레딧** |
| 평균 페이지 | **2 크레딧** |
| 빽빽한 페이지 (~2000자 초과) | **3~4 크레딧** |

평균 페이지 기준 1만원 = 약 150장 분석 (사용자 의도와 일치).

### 핵심 원칙 — "모르는 게 약" UX

- 사용자에게 **정확한 단가 비공개**
- **잔액만 명확히 표시** (충전 시점 결정에 필수)
- 차감 후 **사후 토스트로만 가벼운 안내** ("이 페이지: 2 크레딧 차감 / 잔액 N")
- 운영자가 텍스트 양·토큰 비용·마진을 자유롭게 조정 가능

### 채택 배경

- 페이지 단위 정액 (1페이지 = 1 크레딧) 검토 → **"초보용 책 텍스트 적은데 1콩이야?"** 사용자 불공정 인식 우려
- 텍스트 양 단위 (100자 = 1 크레딧) 검토 → 사용자 비용 예측 어려움 + "이번엔 더 차감?" 불쾌감
- 결국 ChatGPT 등의 **토큰 기반 모델 — "사용자가 정확히 모르는" 패턴**이 학습 앱 컨텍스트에 적합하다고 판단
- → 사용자는 잔액만 보고 충전 결정, 차감은 시스템 자동·불투명

### 신뢰 보호 가드

- **사후 토스트**: 차감 직후 잠깐 "이 페이지: 2 크레딧 / 잔액 98" 표시 — 매번 따지지는 않게, 보고 싶으면 보게
- **평균 안내 (마케팅·도움말)**: "보통 사진 1장당 1~4 크레딧" 정도의 대략 가이드. 정확 단가 표는 비공개

## 4. 원가 분석 (Gemini 2.5 Flash 기준)

### 처리 흐름

`reading.html` 의 메인 경로 = 2-phase 파이프라인:
1. **Phase 1 — `callGeminiReadingOcr`**: 이미지 → 원문 텍스트
2. **Phase 2 — `callGeminiReadingFinalize`**: 텍스트 → 번역 + 학습단어

### 평균 페이지 (소설 1장, ~150 단어, ~800자)

| 단계 | Input | Output + Thinking | 비용 |
|---|---|---|---|
| Phase 1 (OCR) | 이미지 ~600 + 프롬프트 ~400 = 1,000 tok | OCR ~200 + thinking ~500 = 700 tok | ~$0.00029 |
| Phase 2 (Finalize) | 프롬프트 ~600 + 본문 ~250 = 850 tok | 번역 ~250 + 학습단어 ~250 + thinking ~500 = 1,000 tok | ~$0.00036 |
| **합계** | | | **~$0.00065 (약 0.9원)** |

가격 기준: input $0.075 / output $0.30 per 1M tokens, 1$ = 1,400원 가정

### 변동 폭

- 평균 페이지: **약 1~2원**
- 빽빽한 페이지: **약 3~5원** (5배)
- 극단 케이스 (사전·백과사전): **약 5~10원** (10배)

### 마진 계산

1 페이지당 67원 (1만원 패키지 기준) 가정:

| 케이스 | 원가 | 가격 | 마진 |
|---|---|---|---|
| 평균 페이지 | ~1~2원 | 67원 | **30~60배** ✅ |
| 빽빽한 페이지 | ~3~5원 | 67원 | 13~22배 ✅ |
| 극단 케이스 | ~10원 | 67원 | 6배 ✅ |

→ 평균적으로 매우 안정적, 극단 케이스에도 마진 6배 이상.

- ✅ 단위 이름: **크레딧 (🪙)**
- ✅ 충전 옵션: 3종 (5천/1만/2만)
- ✅ 차감 정책 골격: 옵션 B (1~4 크레딧, 자동·불투명)

## 6. 미해결 결정 포인트 (구현 전 확정 필요)

### 6.2 차감 정책 임계값

글자수 정확한 기준 결정 필요:
- 1 크레딧 ↔ 2 크레딧 임계값 (예: 500자, 700자, 1000자 중)
- 2 크레딧 ↔ 3 크레딧 임계값
- 3 크레딧 ↔ 4 크레딧 임계값

운영 시작 후 실제 페이지 분포 데이터 보고 조정 가능.

### 6.4 DB 스키마

- `user_usage` 테이블에 `credit_balance` 컬럼 추가
- `credit_transactions` 별도 테이블? (충전·차감 이력 — 환불·문의 대응)

### 6.5 Edge Function

- 충전 처리: 결제 webhook → credit_balance 증가 RPC
- 차감 처리: Reading 분석 시작 시 잔액 확인 → 분석 후 차감 RPC
- RLS 정책: 본인 잔액만 조회 가능, 충전·차감은 SECURITY DEFINER RPC 로만
